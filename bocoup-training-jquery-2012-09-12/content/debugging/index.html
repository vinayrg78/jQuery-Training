Exception handling

An exception will be thrown when this button is clicked, because the jQuery .html() method isn't available on DOM nodes. When we encounter issues like this, we can tell the debugger to pause on exceptions (we should tell it to pause on uncaught exceptions). Then, the debugger will jump to the line of code where the exception is thrown the next time it happens.

The debugger statement

If we want to force our code to pause even when there isn't an error, we can insert a debugger statement. This will make code execution pause when the debugger reaches this line of code; at that point, you can inspect anything of interest to you, and then allow the code to resume if you'd like.

Setting breakpoints

Sometimes it's not practical (or possible) to modify your JavaScript files to add a debugger statement; sometimes you're trying to debug a page that can't be reloaded because state will be lost. In those cases, you can set a breakpoint manually in the developer tools "Sources" pane. Once set, breakpoints will stop code execution as if the debugger statement was used, but can be bypassed or disabled without reloading the page.

Debugging jQuery errors

Debugging skills are especially useful when you run into an error that appears to be coming from jQuery itself. Sometimes the console will report that jQuery is throwing an exception when the error is really in your code. In these cases, utilize both the function "Call Stack" and the "Pretty Print" debugger tools to find out where the error originates. Also, it's always helpful to debug against the unminified jQuery as it is much easier to example.
